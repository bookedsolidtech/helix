The npm installation approach integrates HELIX web components directly into your Drupal theme's build pipeline, giving you complete control over bundling, optimization, and version management. This is the **recommended approach for production custom themes** where performance, security, and maintainability are critical.

This comprehensive guide covers the entire npm workflow—from initial installation through build configuration, asset generation, library definition, and production deployment strategies. By the end, you'll have a production-ready Drupal theme with optimized HELIX component loading.

---

## Why Choose NPM Installation

### The npm Advantage

Unlike CDN installation (which relies on external services) or module approaches (which add Drupal-specific abstraction layers), npm integration puts you in the driver's seat. You control exactly which components ship to browsers, how they're bundled, when they're updated, and how they're optimized.

**Core benefits:**

1. **Tree-Shaking**: Import only the components you use. If your theme uses 5 of 20 components, only those 5 ship to production—resulting in 60-80% smaller bundles compared to full library loading.

2. **Build-Time Optimization**: Modern bundlers (Vite, Webpack, Rollup) apply advanced optimizations at build time: minification, code splitting, compression, dead code elimination, and scope hoisting.

3. **Version Control**: `package-lock.json` guarantees identical library versions across development, staging, and production. No surprises from CDN auto-updates or breaking changes slipping through.

4. **Offline Development**: Once installed, developers work entirely offline. No network dependency, no CDN outages disrupting local work.

5. **Bundle Integration**: HELIX components bundle with your theme's JavaScript, eliminating separate HTTP requests and leveraging shared dependencies.

6. **Security Posture**: No external runtime dependencies. All code is reviewed, installed locally, and served from your origin servers—critical for HIPAA-compliant healthcare applications.

7. **Asset Aggregation**: Drupal can aggregate your built JavaScript with other theme/module scripts, reducing HTTP requests and improving cache efficiency.

8. **Custom Workflows**: Combine HELIX with other npm packages (date pickers, charting libraries, form validators) in optimized bundles tailored to your application.

### When to Use npm

**Ideal for:**

- Production custom themes for healthcare organizations
- Performance-critical applications (target Lighthouse scores >95)
- Sites requiring HIPAA compliance or strict security audits
- Teams with front-end build expertise (React, Vue, Angular backgrounds)
- Projects with CI/CD pipelines supporting Node.js build steps
- Multi-environment workflows (dev/staging/production)
- Themes consuming multiple component libraries or design systems

**Not ideal for:**

- Quick prototypes or proof-of-concept work (use CDN)
- Small sites without existing build pipelines (use CDN or module)
- Teams without JavaScript build tool experience (consider module approach)
- Contrib themes you don't maintain (use module or CDN)

## Prerequisites

Before beginning npm installation, ensure your development environment meets these requirements:

### System Requirements

**Node.js** (version 20+):

```bash
node --version  # Should output v20.0.0 or higher
```

**npm** (version 10+):

```bash
npm --version   # Should output 10.0.0 or higher
```

**Drupal** (version 10 or 11):

```bash
drush status | grep "Drupal version"
```

**Custom Theme**: You must have a custom Drupal theme (contrib themes don't support custom npm builds).

### Installing Node.js

**macOS** (via Homebrew):

```bash
brew install node@20
```

**Ubuntu/Debian**:

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs
```

**Windows**:
Download from [nodejs.org](https://nodejs.org) and install the LTS version.

**Verify installation**:

```bash
node --version && npm --version
```

### Theme Directory Structure

Your custom theme should follow this structure:

```
web/themes/custom/mytheme/
├── mytheme.info.yml       # Theme definition
├── mytheme.libraries.yml  # Asset library definitions
├── templates/             # TWIG template overrides
├── package.json           # npm dependencies (YOU WILL CREATE THIS)
├── src/                   # Source files (YOU WILL CREATE THIS)
│   └── js/
│       └── theme.js       # Main JavaScript entry point
└── dist/                  # Built assets (GENERATED BY BUILD TOOLS)
    └── js/
        └── theme.js       # Compiled, optimized bundle
```

If you don't have a custom theme yet, create one:

```bash
cd web/themes/custom
mkdir mytheme
cd mytheme
```

**Minimal `mytheme.info.yml`**:

```yaml
name: My Theme
type: theme
core_version_requirement: ^10 || ^11
base theme: stable9
```

## Step 1: Initialize npm

Navigate to your theme directory and initialize npm:

```bash
cd web/themes/custom/mytheme
npm init -y
```

This creates `package.json`:

```json
{
  "name": "mytheme",
  "version": "1.0.0",
  "description": "Custom Drupal theme with HELIX components",
  "type": "module",
  "scripts": {},
  "keywords": [],
  "author": "",
  "license": "GPL-2.0-or-later"
}
```

### Configure Package Type

**Critical**: Set `"type": "module"` to enable ES module support:

```json
{
  "name": "mytheme",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "dev": "vite build --watch",
    "build": "vite build",
    "prod": "vite build --mode production"
  }
}
```

## Step 2: Install HELIX Library

Install the HELIX component library as a production dependency:

```bash
npm install @helixui/library --save
```

This adds HELIX to `package.json`:

```json
{
  "dependencies": {
    "@helixui/library": "^0.0.1"
  }
}
```

### Version Pinning Strategy

**Production environments** should use **exact versions** (no `^` or `~` prefixes):

```bash
npm install @helixui/library --save-exact
```

Resulting `package.json`:

```json
{
  "dependencies": {
    "@helixui/library": "0.0.1"
  }
}
```

**Why pin exact versions?**

- Prevents unexpected breaking changes from semver-compatible updates
- Ensures reproducible builds across all environments
- Critical for healthcare/regulated industries requiring change control
- Facilitates debugging (exact version = exact code)

**Development environments** can use caret ranges for convenience:

```json
{
  "dependencies": {
    "@helixui/library": "^0.0.1" // Allows 0.0.2, 0.0.3, etc.
  }
}
```

### Verify Installation

Check that HELIX installed correctly:

```bash
npm list @helixui/library
```

Output:

```
mytheme@1.0.0 /path/to/mytheme
└── @helixui/library@0.0.1
```

Inspect installed files:

```bash
ls -la node_modules/@helixui/library/
```

You should see:

```
dist/              # Compiled component bundles
src/               # Source TypeScript files
custom-elements.json  # Component API documentation
package.json       # Library package definition
```

## Step 3: Choose a Build Tool

HELIX components are distributed as ES modules. To bundle them into your theme, you need a build tool. The two most popular options are **Vite** (modern, fast) and **Webpack** (mature, ubiquitous).

### Option A: Vite (Recommended)

**Why Vite:**

- Extremely fast builds (10-100x faster than Webpack for development)
- Zero-config ES module support
- Built-in TypeScript support
- Optimized for Lit web components
- Modern developer experience with HMR (Hot Module Replacement)

**Install Vite**:

```bash
npm install vite --save-dev
```

**Create Vite config** (`vite.config.js`):

```javascript
import { defineConfig } from 'vite';
import { resolve } from 'path';

export default defineConfig({
  build: {
    // Output directory for built assets
    outDir: 'dist/js',

    // Library mode (builds as ES module)
    lib: {
      entry: resolve(__dirname, 'src/js/theme.js'),
      formats: ['es'],
      fileName: 'theme',
    },

    // Rollup-specific options
    rollupOptions: {
      output: {
        // Preserve module structure for better debugging
        preserveModules: false,

        // Manual chunk splitting for better caching
        manualChunks: {
          helix: ['@helixui/library'],
        },
      },
    },

    // Minification
    minify: 'terser',
    terserOptions: {
      compress: {
        drop_console: true, // Remove console.log in production
        drop_debugger: true,
      },
    },

    // Source maps for debugging
    sourcemap: process.env.NODE_ENV === 'production' ? false : true,
  },
});
```

**Update `package.json` scripts**:

```json
{
  "scripts": {
    "dev": "vite build --watch",
    "build": "vite build",
    "prod": "NODE_ENV=production vite build"
  }
}
```

### Option B: Webpack

**Why Webpack:**

- Mature ecosystem with extensive plugin support
- More control over advanced optimization scenarios
- Familiar to teams already using Webpack
- Better integration with legacy build pipelines

**Install Webpack**:

```bash
npm install webpack webpack-cli --save-dev
```

**Create Webpack config** (`webpack.config.js`):

```javascript
import path from 'path';
import { fileURLToPath } from 'url';

const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);

export default {
  mode: process.env.NODE_ENV === 'production' ? 'production' : 'development',

  entry: './src/js/theme.js',

  output: {
    filename: 'theme.js',
    path: path.resolve(__dirname, 'dist/js'),
    module: true, // Output as ES module
  },

  experiments: {
    outputModule: true, // Enable ES module output
  },

  optimization: {
    minimize: process.env.NODE_ENV === 'production',
    splitChunks: {
      cacheGroups: {
        helix: {
          test: /[\\/]node_modules[\\/]@helix[\\/]/,
          name: 'helix',
          chunks: 'all',
        },
      },
    },
  },

  devtool: process.env.NODE_ENV === 'production' ? false : 'source-map',
};
```

**Update `package.json` scripts**:

```json
{
  "scripts": {
    "dev": "webpack --watch",
    "build": "webpack",
    "prod": "NODE_ENV=production webpack"
  }
}
```

## Step 4: Import HELIX Components

Create your theme's JavaScript entry point and import HELIX components.

### Full Library Import (Simplest)

**File**: `src/js/theme.js`

```javascript
/**
 * Theme JavaScript Entry Point
 * Imports HELIX web components library
 */

// Import entire HELIX library
import '@helixui/library';

// Custom theme JavaScript
console.log('[Theme] HELIX components loaded');

// Example: Listen for HELIX component events
document.addEventListener('hx-button-click', (e) => {
  console.log('Button clicked:', e.detail);
});
```

**Bundle size**: ~45KB minified + gzipped (all 20+ components)

**Use when**: You use most HELIX components across your site.

### Tree-Shaken Import (Performance-Optimized)

**File**: `src/js/theme.js`

```javascript
/**
 * Theme JavaScript Entry Point
 * Tree-shaken HELIX imports (only load what you use)
 */

// Import only specific components
import '@helixui/library/components/hx-button';
import '@helixui/library/components/hx-card';
import '@helixui/library/components/hx-text-input';
import '@helixui/library/components/hx-alert';

// Custom theme JavaScript
console.log('[Theme] HELIX components loaded (tree-shaken)');
```

**Bundle size**: ~15-20KB minified + gzipped (4 components)

**Use when**: You only use a handful of components site-wide.

### Conditional Import (Route-Specific)

For advanced scenarios, load components conditionally based on page type:

**File**: `src/js/theme.js`

```javascript
/**
 * Theme JavaScript Entry Point
 * Conditional component loading based on page type
 */

// Core components (always loaded)
import '@helixui/library/components/hx-button';

// Check page type via data attribute on <body>
const pageType = document.body.dataset.pageType;

if (pageType === 'article') {
  // Article-specific components
  import('@helixui/library/components/hx-card').then(() => {
    console.log('[Theme] Article components loaded');
  });
}

if (pageType === 'form') {
  // Form-specific components
  Promise.all([
    import('@helixui/library/components/hx-text-input'),
    import('@helixui/library/components/hx-textarea'),
    import('@helixui/library/components/hx-select'),
    import('@helixui/library/components/hx-checkbox'),
  ]).then(() => {
    console.log('[Theme] Form components loaded');
  });
}
```

**TWIG template** (set page type):

```twig
{# templates/html.html.twig #}
<body{{ attributes.setAttribute('data-page-type', node.bundle) }}>
  {{ page }}
</body>
```

**Result**: Pages only download the components they actually render.

## Step 5: Build Assets

Run the build tool to compile HELIX components into your theme's JavaScript bundle.

### Development Build

```bash
npm run dev
```

**What happens:**

1. Vite/Webpack reads `src/js/theme.js`
2. Resolves `@helixui/library` imports from `node_modules/`
3. Bundles components + dependencies
4. Outputs to `dist/js/theme.js`
5. Watches for file changes and rebuilds automatically

**File structure after build**:

```
web/themes/custom/mytheme/
├── dist/
│   └── js/
│       ├── theme.js           # Your code + HELIX components
│       ├── theme.js.map       # Source map (dev only)
│       └── helix.js           # HELIX chunk (if code-split)
├── src/
│   └── js/
│       └── theme.js
├── package.json
└── vite.config.js
```

### Production Build

```bash
npm run prod
```

**What happens:**

1. Same as dev build, BUT:
2. Minification enabled (removes whitespace, shortens variable names)
3. Dead code elimination (removes unused exports)
4. Tree-shaking (removes unused imports)
5. Console statements removed
6. Source maps disabled
7. Compression applied

**Result**: `dist/js/theme.js` is 70-80% smaller than dev build.

**Example sizes**:

- Development build: 180KB
- Production build: 45KB (minified + gzipped: 18KB)

### Verify Build Output

Check that files built successfully:

```bash
ls -lh dist/js/
```

Output:

```
-rw-r--r--  1 user  staff   18K  theme.js
```

Inspect bundle contents (optional):

```bash
head -n 20 dist/js/theme.js
```

You should see minified JavaScript with no human-readable spacing (production) or readable code with whitespace (development).

## Step 6: Define Drupal Library

Create a Drupal library definition that references your built assets.

### Basic Library Definition

**File**: `mytheme.libraries.yml`

```yaml
# Global library (loaded on all pages)
global:
  version: VERSION
  js:
    dist/js/theme.js:
      attributes:
        type: module
      preprocess: false
  dependencies:
    - core/once
```

**Explanation:**

- **`global`**: Library machine name (can be anything; `global` is conventional)
- **`version: VERSION`**: Drupal replaces `VERSION` with the theme's version from `.info.yml`, providing automatic cache busting
- **`dist/js/theme.js`**: Path relative to theme root
- **`type: module`**: Critical! Tells browsers this is an ES module
- **`preprocess: false`**: Disables Drupal's asset aggregation (ES modules don't aggregate well)
- **`core/once`**: Dependency on Drupal's `once()` utility (useful for Drupal Behaviors)

### Alternative: Manual Version Pinning

If `VERSION` placeholder doesn't suit your workflow, use explicit versions:

```yaml
global:
  version: 1.2.4 # Manually increment on each release
  js:
    dist/js/theme.js:
      attributes:
        type: module
      preprocess: false
```

**When to increment:**

- After `npm run build` (production builds)
- After updating `@helixui/library` version
- After any code changes to `src/js/theme.js`

### Multi-Bundle Approach (Code Splitting)

If your build produces multiple files (e.g., `theme.js` + `helix.js` chunk):

```yaml
global:
  version: VERSION
  js:
    dist/js/theme.js:
      attributes:
        type: module
      preprocess: false
    dist/js/helix.js:
      attributes:
        type: module
      preprocess: false
  dependencies:
    - core/once
```

**Note**: Order matters. Load dependencies first.

### Per-Component Libraries (Advanced)

For maximum flexibility, define separate libraries per component group:

```yaml
# Core components (always loaded)
core-components:
  version: VERSION
  js:
    dist/js/core.js:
      attributes:
        type: module

# Form components (loaded only on forms)
form-components:
  version: VERSION
  js:
    dist/js/forms.js:
      attributes:
        type: module
  dependencies:
    - mytheme/core-components

# Article components (loaded only on article pages)
article-components:
  version: VERSION
  js:
    dist/js/articles.js:
      attributes:
        type: module
  dependencies:
    - mytheme/core-components
```

**Build configuration** (Vite with multiple entry points):

```javascript
// vite.config.js
export default defineConfig({
  build: {
    rollupOptions: {
      input: {
        core: resolve(__dirname, 'src/js/core.js'),
        forms: resolve(__dirname, 'src/js/forms.js'),
        articles: resolve(__dirname, 'src/js/articles.js'),
      },
    },
  },
});
```

## Step 7: Attach Library to Theme

Make the library available to your theme.

### Global Attachment (All Pages)

**File**: `mytheme.info.yml`

```yaml
name: My Theme
type: theme
core_version_requirement: ^10 || ^11
base theme: stable9

libraries:
  - mytheme/global # Loads on every page
```

**Result**: All TWIG templates and Drupal Behaviors have access to HELIX components.

### Conditional Attachment (Specific Templates)

For performance-sensitive sites, attach libraries only where needed:

**File**: `templates/node--article.html.twig`

```twig
{# Attach library only on Article nodes #}
{{ attach_library('mytheme/article-components') }}

<article{{ attributes }}>
  <hx-card variant="featured" elevation="raised">
    <span slot="heading">{{ node.title.value }}</span>
    {{ content.body }}
  </hx-card>
</article>
```

**File**: `templates/page.html.twig`

```twig
{# Attach form library on form pages #}
{% if page_type == 'webform' %}
  {{ attach_library('mytheme/form-components') }}
{% endif %}

{{ page.content }}
```

### Programmatic Attachment (PHP)

Attach libraries via PHP hooks for complex logic:

**File**: `mytheme.theme`

```php
<?php

/**
 * Implements hook_page_attachments().
 */
function mytheme_page_attachments(array &$attachments) {
  // Attach HELIX library on all pages except admin
  if (!\Drupal::service('router.admin_context')->isAdminRoute()) {
    $attachments['#attached']['library'][] = 'mytheme/global';
  }
}

/**
 * Implements hook_preprocess_node().
 */
function mytheme_preprocess_node(&$variables) {
  // Attach article components only on article nodes
  if ($variables['node']->bundle() === 'article') {
    $variables['#attached']['library'][] = 'mytheme/article-components';
  }
}
```

## Asset Generation Strategies

### Strategy 1: Single Bundle (Default)

**Build output**: 1 file (`theme.js`)

**Vite config**:

```javascript
export default defineConfig({
  build: {
    lib: {
      entry: resolve(__dirname, 'src/js/theme.js'),
      formats: ['es'],
      fileName: 'theme',
    },
  },
});
```

**Pros:**

- Simple deployment (1 file to manage)
- Fewer HTTP requests
- Easier cache management

**Cons:**

- Larger initial download
- Can't selectively load components

**Use when:** Most pages use most components.

### Strategy 2: Code Splitting (Performance)

**Build output**: Multiple files (`theme.js`, `helix.js`, `vendors.js`)

**Vite config**:

```javascript
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          helix: ['@helixui/library'],
          vendors: ['lit'],
        },
      },
    },
  },
});
```

**Pros:**

- Better browser caching (vendors chunk rarely changes)
- Parallel download (HTTP/2)
- Smaller per-file sizes

**Cons:**

- More complex deployment
- More HTTP requests (mitigated by HTTP/2)

**Use when:** Large applications with multiple pages.

### Strategy 3: Per-Component Bundles (Maximum Flexibility)

**Build output**: 1 file per component (`hx-button.js`, `hx-card.js`, etc.)

**Vite config**:

```javascript
export default defineConfig({
  build: {
    rollupOptions: {
      input: {
        'hx-button': 'node_modules/@helixui/library/src/components/hx-button/index.js',
        'hx-card': 'node_modules/@helixui/library/src/components/hx-card/index.js',
        'hx-text-input': 'node_modules/@helixui/library/src/components/hx-text-input/index.js',
      },
      output: {
        entryFileNames: 'components/[name].js',
      },
    },
  },
});
```

**Drupal library**:

```yaml
hx-button:
  version: VERSION
  js:
    dist/js/components/hx-button.js:
      attributes:
        type: module

hx-card:
  version: VERSION
  js:
    dist/js/components/hx-card.js:
      attributes:
        type: module
```

**Pros:**

- Perfect tree-shaking (only load exactly what renders)
- Maximum performance
- Granular cache control

**Cons:**

- Complex build configuration
- Many library definitions to maintain
- Many HTTP requests (mitigated by HTTP/2)

**Use when:** Large sites with varied page types.

## Version Management

### Updating HELIX Library

When HELIX releases a new version, update via npm:

**Check current version**:

```bash
npm list @helixui/library
```

**Check available versions**:

```bash
npm view @helixui/library versions
```

**Update to latest patch version** (e.g., 0.0.1 → 0.0.2):

```bash
npm update @helixui/library
```

**Update to specific version**:

```bash
npm install @helixui/library@0.1.0 --save-exact
```

**Rebuild assets**:

```bash
npm run build
```

**Increment Drupal library version** (`mytheme.libraries.yml`):

```yaml
global:
  version: 1.2.5 # Was 1.2.4
```

**Clear Drupal caches**:

```bash
drush cr
```

### Lockfile Management

**`package-lock.json`** ensures reproducible installs:

**Commit to version control**:

```bash
git add package-lock.json
git commit -m "Update HELIX library to 0.1.0"
```

**When collaborating**:

- After pulling code with updated `package-lock.json`, run `npm ci` (clean install)
- Never manually edit `package-lock.json`

**CI/CD environments** should use:

```bash
npm ci  # Not npm install
```

`npm ci` installs exact versions from lockfile, ensuring build reproducibility.

## Development vs Production Builds

### Development Workflow

**Start watch mode**:

```bash
npm run dev
```

**What happens:**

1. Initial build completes
2. Vite/Webpack watches `src/` for changes
3. On file save, rebuilds automatically
4. Browser auto-refreshes (if using LiveReload or similar)

**Development build characteristics:**

- Source maps enabled (easy debugging)
- Console statements preserved
- Readable variable names
- No minification
- Fast rebuild times (100-300ms)

**Use during**: Local development, debugging, testing.

### Production Workflow

**Run production build**:

```bash
npm run prod
```

**What happens:**

1. Build runs in production mode
2. Minification applied
3. Tree-shaking eliminates dead code
4. Console statements removed
5. Source maps disabled
6. Output optimized for size + performance

**Production build characteristics:**

- 70-80% smaller than dev builds
- No source maps (or external source maps)
- Minified, obfuscated code
- All console.log removed
- Slower build times (1-3 seconds)

**Use during**: Staging deployments, production releases.

### Environment-Specific Configuration

**Use environment variables** to customize builds:

**Vite example**:

```javascript
// vite.config.js
export default defineConfig({
  build: {
    minify: process.env.NODE_ENV === 'production' ? 'terser' : false,
    sourcemap: process.env.NODE_ENV !== 'production',
  },
});
```

**Set environment**:

```bash
NODE_ENV=production npm run build
```

**Or update `package.json`**:

```json
{
  "scripts": {
    "dev": "vite build --watch",
    "build": "vite build",
    "prod": "cross-env NODE_ENV=production vite build"
  },
  "devDependencies": {
    "cross-env": "^7.0.3"
  }
}
```

## Advantages Over CDN

### 1. Complete Control

**CDN**: You're at the mercy of external services. If jsDelivr goes down, your components fail.

**npm**: All assets served from your origin. Uptime = your hosting uptime.

### 2. Tree-Shaking

**CDN**: Load full bundle (48KB) even if you only use 3 components.

**npm**: Build includes only imported components (15KB for 3 components = 68% savings).

### 3. Security & Compliance

**CDN**: External dependency violates HIPAA/privacy regulations in healthcare contexts.

**npm**: All code reviewed, installed locally, served internally. Passes security audits.

### 4. Performance Consistency

**CDN**: Performance varies by geography, CDN health, network conditions.

**npm**: Served from your edge (if using CDN for static assets) or origin with predictable latency.

### 5. Offline Development

**CDN**: Developers need internet access to work.

**npm**: Full offline development after initial install.

### 6. Custom Bundling

**CDN**: Can't combine HELIX with other libraries in optimized bundles.

**npm**: Bundle HELIX with date pickers, validators, charting libraries, etc., for perfect cache alignment.

### 7. No Privacy Leaks

**CDN**: User IP addresses, user agents, referers sent to third-party CDN.

**npm**: Zero external requests. Patient data never leaves your infrastructure.

```
└── templates/
    ├── html.html.twig

    └── node--article.html.twig
```

**`package.json`**:

```json
{
  "name": "healthsystem-theme",
  "version": "1.0.0",
  "type": "module",
  "description": "Hospital system theme with HELIX components",
  "scripts": {
    "dev": "vite build --watch",
    "build": "vite build",
    "prod": "cross-env NODE_ENV=production vite build"
  },
  "dependencies": {
    "@helixui/library": "0.0.1"
  },
  "devDependencies": {
    "cross-env": "^7.0.3",
    "vite": "^6.2.0"
  }
}
```

**`vite.config.js`**:

```javascript
import { defineConfig } from 'vite';
import { resolve } from 'path';

export default defineConfig({
  build: {
    outDir: 'dist/js',
    lib: {
      entry: resolve(__dirname, 'src/js/theme.js'),
      formats: ['es'],
      fileName: 'theme',
    },
    rollupOptions: {
      output: {
        manualChunks: {
          helix: ['@helixui/library'],
        },
      },
    },
    minify: process.env.NODE_ENV === 'production' ? 'terser' : false,
    sourcemap: process.env.NODE_ENV !== 'production',
  },
});
```

**`src/js/theme.js`**:

```javascript
/**
 * Health System Theme - Main JavaScript
 */

// Import HELIX components (tree-shaken)
import '@helixui/library/components/hx-button';
import '@helixui/library/components/hx-card';
import '@helixui/library/components/hx-alert';
import '@helixui/library/components/hx-text-input';
import '@helixui/library/components/hx-textarea';
import '@helixui/library/components/hx-select';

// Drupal behaviors for HELIX integration
import './behaviors/helix-forms.js';

console.log('[HealthSystem] Theme loaded with HELIX components');
```

**`src/js/behaviors/helix-forms.js`**:

```javascript
/**
 * Drupal Behavior: HELIX Form Integration
 */
(function (Drupal, once) {
  Drupal.behaviors.helixForms = {
    attach(context) {
      // Initialize HELIX form components
      once('helix-form-init', 'hx-text-input', context).forEach((input) => {
        input.addEventListener('hx-input', (e) => {
          console.log('Input changed:', e.detail.value);
        });
      });

      // Handle form submissions
      once('helix-form-submit', 'hx-form', context).forEach((form) => {
        form.addEventListener('hx-submit', async (e) => {
          e.preventDefault();

          // Custom form handling
          const formData = e.detail.formData;
          console.log('Form submitted:', formData);
        });
      });
    },
  };
})(Drupal, once);
```

**`healthsystem.libraries.yml`**:

```yaml
global:
  version: VERSION
  js:
    dist/js/theme.js:
      attributes:
        type: module
      preprocess: false
    dist/js/helix.js:
      attributes:
        type: module
      preprocess: false
  dependencies:
    - core/once
    - core/drupal
```

**`healthsystem.info.yml`**:

```yaml
name: Health System Theme
```

## Best Practices

### 1. Always Use Exact Versions in Production

```json
{
  "dependencies": {
    "@helixui/library": "0.0.1" // Not ^0.0.1 or ~0.0.1
  }
}
```

### 2. Commit package-lock.json

```bash
git add package-lock.json
git commit -m "Lock HELIX library version"
```

### 3. Use CI/CD for Builds

**Never commit built files** (`dist/`). Build on deployment:

**GitHub Actions example**:

```yaml
# .github/workflows/deploy.yml
name: Deploy Theme
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: 20
      - run: npm ci
      - run: npm run prod
      - run: rsync -av dist/ ${{ secrets.DEPLOY_PATH }}
```

### 4. Separate Dev/Prod Builds

```json
{
  "scripts": {
    "dev": "vite build --watch",
    "prod": "cross-env NODE_ENV=production vite build"
  }
}
```

### 5. Version Lock Drupal Library

When updating HELIX, increment Drupal library version:

```yaml
global:
  version: 1.2.5 # Was 1.2.4
```

### 6. Tree-Shake Aggressively

Only import components you use:

```javascript
// Bad: Imports everything
import '@helixui/library';

// Good: Imports only what you need
import '@helixui/library/components/hx-button';
import '@helixui/library/components/hx-card';
```

### 7. Monitor Bundle Size

Use bundle analyzers to track size:

```bash
npm install rollup-plugin-visualizer --save-dev
```

**Vite config**:

```javascript
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({
  plugins: [visualizer({ open: true })],
});
```

Run build to see visual bundle breakdown.

### 8. Use Source Maps Wisely

**Development**: Enable for debugging.
**Production**: Disable (or use external source maps).

```javascript
export default defineConfig({
  build: {
    sourcemap: process.env.NODE_ENV !== 'production',
  },
});
```

## Troubleshooting

### Issue: "Cannot find module '@helixui/library'"

**Cause**: Package not installed or wrong path.

**Fix**:

```bash
npm install @helixui/library --save
```

Verify install:

```bash
ls node_modules/@helixui/library
```

### Issue: Components don't render (no Shadow DOM)

**Cause**: Library not loaded or wrong script type.

**Fix**: Check `libraries.yml` has `type: module`:

```yaml
global:
  js:
    dist/js/theme.js:
      attributes:
        type: module # CRITICAL
```

### Issue: Build fails with "ReferenceError: \_\_dirname is not defined"

**Cause**: Using CommonJS `__dirname` in ES modules.

**Fix**:

```javascript
import { fileURLToPath } from 'url';
import path from 'path';

const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
```

### Issue: Drupal serves old JavaScript (cache issue)

**Cause**: Browser or Drupal cache.

**Fix**:

```bash
# Clear Drupal cache
drush cr

# Increment library version
# mytheme.libraries.yml
global:
  version: 1.2.5  # Was 1.2.4

# Hard reload browser (Cmd+Shift+R)
```

### Issue: Bundle size too large

**Cause**: Importing full library instead of tree-shaking.

**Fix**: Import specific components:

```javascript
// Before (48KB)
import '@helixui/library';

// After (18KB)
import '@helixui/library/components/hx-button';
import '@helixui/library/components/hx-card';
```

### Issue: "exports is not defined" error

**Cause**: Build output is CommonJS, but Drupal expects ES module.

**Fix**: Ensure build config outputs ES modules:

**Vite**:

```javascript
lib: {
  formats: ['es'],  // Not 'umd' or 'cjs'
}
```

**Webpack**:

```javascript
experiments: {
  outputModule: true,
}
```

## Summary

npm installation provides the optimal balance of control, performance, and security for production Drupal themes:

**Key Steps Recap**:

1. Initialize npm in theme directory (`npm init -y`)
2. Install HELIX library (`npm install @helixui/library --save-exact`)
3. Choose build tool (Vite recommended)
4. Import components in `src/js/theme.js`
5. Build assets (`npm run build`)
6. Define Drupal library in `mytheme.libraries.yml`
7. Attach library via `.info.yml` or TWIG
8. Clear Drupal caches (`drush cr`)

**Production Checklist**:

- [ ] Exact version pinned in `package.json`
- [ ] `package-lock.json` committed to version control
- [ ] Tree-shaking enabled (import only used components)
- [ ] Production build script configured (`npm run prod`)
- [ ] CI/CD pipeline builds assets on deployment
- [ ] Drupal library `type: module` attribute present
- [ ] Source maps disabled for production
- [ ] Bundle size monitored (<20KB recommended)
