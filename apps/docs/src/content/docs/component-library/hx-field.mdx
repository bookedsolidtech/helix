## Overview

`hx-field` is a layout wrapper that coordinates the four structural elements of any form field: label, form control, help text, and error message. It is designed to wrap any native HTML form element (`<input>`, `<textarea>`, `<select>`) or Helix custom element and provide consistent spacing, labeling, and accessibility plumbing across the shadow DOM boundary.

**Use `hx-field` when:**

- You need consistent label + control + help text + error layout in a form
- You are wrapping a native HTML input element that lacks built-in Helix field structure
- You need to coordinate `aria-label`, `aria-required`, `aria-invalid`, and `aria-describedby` on a slotted control without managing them manually
- You want a single token-driven API to control field-level spacing, label color, and error color across a form

**Do not use `hx-field`** as a form control itself — it is a pure layout wrapper and is not form-associated. Use it to wrap controls, not to replace them. For Helix native controls (`hx-text-input`, etc.) that already handle their own ARIA and labeling, check whether the native component already includes built-in field structure before wrapping it in `hx-field`.

**Note on the light DOM side effect:** `hx-field` injects a visually-hidden `<span>` into its light DOM children on connection. This span (identifiable by its `id` ending in `-desc`) serves as the `aria-describedby` anchor for the slotted control, bridging the shadow DOM accessibility boundary. It is removed on disconnect. Do not remove or modify this span.


## Properties

| Property   | Attribute   | Type                   | Default    | Description                                                                                                                                  |
| ---------- | ----------- | ---------------------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `label`    | `label`     | `string`               | `''`       | The visible label text. When set, a `<label>` element is rendered in the shadow root and `aria-label` is applied to the slotted control.     |
| `required` | `required`  | `boolean`              | `false`    | When true, shows a required asterisk on the label and sets `aria-required="true"` on the slotted control. Reflects to host attribute.        |
| `error`    | `error`     | `string`               | `''`       | Error message text. When set, renders a `role="alert"` container, adds `aria-invalid="true"` to the slotted control, and hides help text.    |
| `helpText` | `help-text` | `string`               | `''`       | Help text displayed below the control. Hidden when `error` is active. Announced to AT via `aria-describedby`.                                |
| `disabled` | `disabled`  | `boolean`              | `false`    | Applies visual opacity to the entire field. Does not disable the slotted control — set `disabled` on the control directly. Reflects to host. |
| `hxSize`   | `hx-size`   | `'sm' \| 'md' \| 'lg'` | `'md'`     | Size variant controlling label and help text font sizes. Reflects to host attribute.                                                         |
| `layout`   | `layout`    | `'column' \| 'inline'` | `'column'` | Layout variant. `'column'` stacks label above control; `'inline'` places them side-by-side. Reflects to host attribute.                      |

## Slots

| Slot          | Description                                                                                                                                                           |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| _(default)_   | The form control element — a native `<input>`, `<textarea>`, `<select>`, or custom element. ARIA attributes are bridged to the first form control in this slot.       |
| `label`       | Custom label content. When populated, overrides the `label` property and `aria-label` is removed from the slotted control. Wire up `for`/`id` yourself (see example). |
| `help`        | Custom help text content. When populated, overrides the `helpText` property.                                                                                          |
| `error`       | Custom error content. When populated, the field enters error state (`aria-invalid="true"`, `field--error` class). Add `role="alert"` to slotted error content.        |
| `description` | Additional descriptive content rendered above the control, below the label.                                                                                           |

### Custom Label Slot — for/id Association Required

When using the `label` slot, `aria-label` is NOT automatically applied to the slotted control (because `hx-field` cannot know the ID of your custom label element). You must wire up the association yourself using `for`/`id` — both elements are in the same light DOM so the native HTML association works:

```html
<hx-field>
  <!-- Wire the label to the input via for/id -->
  <label slot="label" for="patient-name-input">
    Patient Name <em style="color: #6b7280;">(legal name)</em>
  </label>
  <input id="patient-name-input" type="text" />
</hx-field>
```

### Custom Error Slot — role="alert" Required

When using the `error` slot, add `role="alert"` to your slotted error content so screen readers announce it when it appears dynamically:

```html
<hx-field>
  <input type="text" />
  <span slot="error" role="alert"> <strong>Error:</strong> This field is required. </span>
</hx-field>
```

## CSS Parts

| Part                 | Description                                                                                         |
| -------------------- | --------------------------------------------------------------------------------------------------- |
| `field`              | The outer flex container wrapping all field segments. Carries state classes (`field--error`, etc.). |
| `label`              | The `<label>` element. Present only when the `label` property is set and the label slot is empty.   |
| `control`            | The wrapper `<div>` around the default slot (slotted form control).                                 |
| `help-text`          | The help text container. Hidden when error is active or neither `helpText` nor help slot is set.    |
| `error-message`      | The error message `<div>`. Present only when the `error` property is set.                           |
| `required-indicator` | The required asterisk `<span>`. Present only when `required` is true and a label is rendered.       |

**Example — part customization:**

```css
/* Style the label to bold in this context */
.my-form hx-field::part(label) {
  font-weight: 700;
  text-transform: uppercase;
  letter-spacing: 0.05em;
}

/* Style the control wrapper */
.my-form hx-field::part(control) {
  border-radius: 0.5rem;
  overflow: hidden;
}
```

## Events

`hx-field` does not emit any custom events. It is a pure layout wrapper. Attach event listeners directly to the slotted control.

```html
<hx-field label="Search Patients">
  <input
    type="search"
    id="patient-search"
    oninput="handleInput(event)"
    onchange="handleChange(event)"
  />
</hx-field>
```

## Accessibility

| Topic                 | Details                                                                                                                                                                                                                                                                |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Label association     | When the `label` property is set, `aria-label` is applied directly to the slotted control, bridging the shadow DOM boundary. The rendered `<label>` is a visual target; focus is delegated via a click handler.                                                        |
| Label slot            | When using the `label` slot, you must wire `for`/`id` between your slotted label and control. Both are in the same light DOM scope so native HTML association works. `aria-label` is not applied in this case.                                                         |
| Required state        | `aria-required="true"` is set on the slotted control when `required` is true. The visual asterisk has `aria-hidden="true"`.                                                                                                                                            |
| Invalid / error state | `aria-invalid="true"` is set on the slotted control when `error` is set or the error slot is populated. The error message container uses `role="alert"` for immediate announcement.                                                                                    |
| Description           | A visually-hidden `<span>` is injected into the component's light DOM (id ending in `-desc`) and referenced via `aria-describedby` on the slotted control. This bridges the shadow DOM IDREF limitation. Error text takes priority over help text in this description. |
| Slot announcer        | A separate `aria-live="assertive"` region in the shadow DOM serves slotted error content. When using the `error` slot, also add `role="alert"` to your content for reliable cross-AT announcement.                                                                     |
| Third-party controls  | ARIA bridging is skipped for `hx-*` elements (they manage their own ARIA). Third-party custom elements can opt out by setting `data-aria-managed` on the element.                                                                                                      |
| Disabled              | `hx-field[disabled]` applies opacity via CSS. The slotted control must have its own `disabled` attribute for interactive disabling.                                                                                                                                    |
| Keyboard navigation   | `hx-field` itself is not a focusable element. All keyboard interaction is handled by the slotted control.                                                                                                                                                              |
| WCAG 2.1 AA           | Verified with axe-core: zero violations across default, required, error, disabled, and all native control variants (input, textarea, select).                                                                                                                          |

## Best Practices

**Do** use `hx-field` for every form input that requires a visible label, help text, or validation feedback. Consistent wrapping ensures uniform spacing and accessible labeling across your form.

**Do** set `error` programmatically on validation failure rather than toggling visibility of custom error elements. The built-in `role="alert"` ensures screen readers announce errors immediately.

**Do** add `disabled` to both `hx-field` and the slotted control when disabling a field. `hx-field[disabled]` applies visual treatment; the control's own `disabled` attribute prevents interaction.

**Do not** nest `hx-field` inside another `hx-field`. Each field should be a single flat wrapper.

**Do not** use `hx-field` for checkbox groups or radio groups where multiple controls share one label — use a `<fieldset>`/`<legend>` pattern instead and consider `hx-checkbox-group` or `hx-radio-group`.

**Do not** modify or remove the visually-hidden `<span>` injected into `hx-field`'s light DOM (identifiable by `id` ending in `-desc`). This span is the `aria-describedby` anchor for the slotted control.

**Do** use the `label` slot only when you need rich label content (icons, custom markup). When you do, wire `for`/`id` between your label element and the slotted control — `aria-label` is not applied automatically in this case.

**Do** prefer `hx-size="lg"` for patient-facing intake forms to improve touch target sizing and readability for users with low vision.
