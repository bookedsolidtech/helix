# HELiX NPM Release Strategy

Canonical reference for publishing `@helixui/library` and `@helixui/tokens` to the npm registry.

---

## Scope & Naming

| Detail | Value |
|--------|-------|
| **npm scope** | `@helix` |
| **Owner** | `himerus` (Jake Strawn) |
| **Existing packages** | `@helixui/ini-parse` (confirms scope ownership) |
| **Packages to publish** | `@helixui/library`, `@helixui/tokens` |
| **Current version** | `0.1.0` (both packages, linked) |

The `@helix` scope is owned and available. HELiX components publish under `@helix` rather than `@bookedsolid` (used by the sister project `helixir`) for a cleaner developer experience. The `@helix` namespace is short, memorable, and accurately represents the project brand.

## Package Architecture

### `@helixui/library`

The full component library: 84 custom elements across 73 component directories, built with Lit 3.x.

**Full library import (tree-shakeable barrel):**

```typescript
import { HelixButton, HelixCard } from '@helixui/library';
```

The barrel is auto-generated by `scripts/generate-barrel.js` and runs as a `prebuild` step. Vite's `discoverEntryPoints()` function auto-discovers per-component entry points from `src/components/hx-*/index.ts`, producing individual chunks. Bundlers (Vite, webpack, Rollup, esbuild) eliminate unused components via tree-shaking.

**Individual component import (surgical):**

```typescript
import { HelixButton } from '@helixui/library/components/hx-button';
```

Supported via the `exports` map in `package.json`. Each component builds to its own entry point at `dist/components/*/index.js`. Use this pattern for micro-frontends or environments where tree-shaking is unavailable.

**CDN / Vanilla HTML (planned -- not yet built):**

```html
<script type="module"
  src="https://cdn.jsdelivr.net/npm/@helixui/library@latest/cdn/hx-button.js">
</script>
```

The CDN entry point is a P1 pre-publish item. It will produce self-contained, dependency-bundled ES modules for each component. This is the primary consumption path for Drupal and vanilla HTML consumers.

**Exports map (current):**

```json
{
  ".": {
    "types": "./dist/index.d.ts",
    "import": "./dist/index.js"
  },
  "./components/*": {
    "types": "./dist/components/*/index.d.ts",
    "import": "./dist/components/*/index.js"
  },
  "./custom-elements.json": "./custom-elements.json"
}
```

### `@helixui/tokens`

Design tokens for theming, available in multiple formats.

```typescript
// Full token set (JS objects + TypeScript types)
import { tokens } from '@helixui/tokens';

// Lit CSS tagged template helpers
import { colorPrimary } from '@helixui/tokens/lit';

// CSS utility helpers
import { getCssVar } from '@helixui/tokens/css';

// Raw CSS custom properties (side-effect import)
import '@helixui/tokens/tokens.css';

// Raw JSON (design tool integration, style-dictionary pipelines)
import tokensJson from '@helixui/tokens/tokens.json';
```

**Exports map (current):**

```json
{
  ".":            { "types": "./dist/index.d.ts",  "import": "./dist/index.js" },
  "./css":        { "types": "./dist/css.d.ts",    "import": "./dist/css.js" },
  "./lit":        { "types": "./dist/lit.d.ts",     "import": "./dist/lit.js" },
  "./utils":      { "types": "./dist/utils.d.ts",   "import": "./dist/utils.js" },
  "./tokens.css": "./dist/tokens.css",
  "./tokens.json": "./src/tokens.json"
}
```

## Recommended Developer Experience

### Install one package, get everything

```bash
npm install @helixui/library
```

That is the entire setup. `@helixui/tokens` is a dependency of `@helixui/library` and installs automatically.

**Why this works:**

1. **Tokens come with components.** Design tokens (`@helixui/tokens`) are declared as a dependency of `@helixui/library`. One install, both packages.
2. **Configuration is included.** The theme provider (`hx-theme`), CSS custom properties, and the full design system activate on import.
3. **Tree-shaking works.** "Install everything, import what you need." Modern bundlers eliminate unused components at build time. A project that imports only `HelixButton` pays only for `HelixButton`.
4. **Sub-components are bundled.** `hx-accordion` includes `hx-accordion-item`. `hx-tabs` includes `hx-tab` and `hx-tab-panel`. 12 sub-components are bundled with their parent directories -- no separate installs required.

### For surgical control (Drupal CDN, micro-frontends)

```html
<!-- Individual component via CDN (future) -->
<script type="module"
  src="https://cdn.jsdelivr.net/npm/@helixui/library/cdn/hx-button.js">
</script>
```

### For token-only consumers (theme providers, design tools)

```bash
npm install @helixui/tokens
```

## Versioning Strategy

### Changesets

Version management uses [Changesets](https://github.com/changesets/changesets). The configuration lives at `.changeset/config.json`.

**Key settings:**

| Setting | Value | Reason |
|---------|-------|--------|
| `linked` | `[["@helixui/library", "@helixui/tokens"]]` | Versions stay in sync across both packages |
| `access` | `restricted` (must change to `public`) | Currently prevents publish; P0 blocker |
| `baseBranch` | `main` | Changesets are consumed on merge to `main` |
| `updateInternalDependencies` | `patch` | `@helixui/library` auto-bumps when `@helixui/tokens` changes |

### Semantic Versioning

- **Major** (1.0.0): Breaking changes to component APIs, removed properties/events, CSS part renames
- **Minor** (0.2.0): New components, new optional properties, new events
- **Patch** (0.1.1): Bug fixes, accessibility improvements, performance improvements

**First publish:** v0.1.1 (patch release with all launch-ready quality fixes applied).

### Adding a Changeset

```bash
npx changeset
# Follow prompts: select packages, bump type, write summary
# Commits a markdown file to .changeset/
```

## Git & Merge Strategy

### Branch Flow

```
feature/* --> dev --> staging --> main
                                  |
                                  v
                          changeset publish
                          npm registry push
                          git tag + release
```

### Merge Policy

- **Merge commits only.** Squash and rebase are disabled on the GitHub repository.
- This matches the `helixir` project, which uses semantic-release with merge commits.
- Merge commits preserve the full history of each feature branch, which is required for accurate changelog generation.

### Promotion Gates

| Transition | Gate | Owner |
|------------|------|-------|
| `feature/* --> dev` | PR review + CI quality gates + CodeRabbit | Automated / Agent |
| `dev --> staging` | All quality gates green | Ava (autonomous) |
| `staging --> main` | Human-in-the-loop approval | Jake (manual) |
| `main --> npm` | CI publish workflow | GitHub Actions (automated) |

The human gate on `staging --> main` is non-negotiable. Every npm publish traces back to an explicit human approval.

## CI/CD Publish Pipeline

### Planned Workflow (`.github/workflows/publish.yml`)

```yaml
name: Publish to npm
on:
  push:
    branches: [main]

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      id-token: write  # Required for npm provenance
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://registry.npmjs.org'

      - run: npm ci
      - run: npm run build
      - run: npm run type-check
      - run: npm run test

      - name: Create Release Pull Request or Publish
        uses: changesets/action@v1
        with:
          publish: npx changeset publish
          version: npx changeset version
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

### Pipeline Sequence

1. PR merges to `main` via the `staging --> main` promotion (human-approved).
2. GitHub Actions triggers on push to `main`.
3. Full quality gate runs: type-check, test suite, build.
4. `changesets/action` detects pending changesets.
5. **If changesets exist:** Opens a "Version Packages" PR that bumps versions and updates CHANGELOGs.
6. **When the Version PR merges:** `changeset publish` pushes to npm, creates git tags, and generates a GitHub Release.

### npm Tokens

- Store `NPM_TOKEN` as a GitHub Actions secret (repository-level).
- Use a granular access token scoped to the `@helix` packages only.
- Enable 2FA for publishing on the npm account (already active on `himerus`).

## Build System Details

### Vite Library Mode

The build uses Vite in library mode with ES module output only. Key configuration:

- **Entry discovery:** `discoverEntryPoints()` scans `src/components/hx-*/index.ts` automatically. Adding a new component directory is all that is needed -- no manual Vite config edits.
- **Externals:** `lit`, `@lit/*`, `@helixui/tokens`, and `@floating-ui/*` are externalized. Consumers provide these as dependencies.
- **Output:** ES modules only (`formats: ['es']`). No CommonJS, no UMD.
- **Sourcemaps:** Enabled for debugging in consumer applications.
- **Type declarations:** Generated by `vite-plugin-dts`, co-located with JS output.
- **Minification:** esbuild (fast, reliable, sufficient for library code).
- **Shared chunks:** Common code is extracted to `dist/shared/` to avoid duplication across component entry points.

### Custom Elements Manifest

Generated by `@custom-elements-manifest/analyzer` after each build. Published as part of the `@helixui/library` package at `custom-elements.json`. Consumed by:

- Storybook (autodocs, controls)
- VS Code custom data (IDE autocompletion)
- Any CEM-compatible tooling

## Decision Log

### Why `@helix` instead of `@bookedsolid`?

| Factor | `@helix` | `@bookedsolid` |
|--------|----------|----------------|
| Length | 6 chars | 11 chars |
| Brand match | Exact | Parent org |
| Available | Yes (owned by himerus) | Yes (used by helixir) |
| Import DX | `@helixui/library` | `@bookedsolid/helix-library` |
| Collision risk | Low (scope is owned) | Low |

The `@helix` scope produces shorter, cleaner imports. The sister project `helixir` already occupies `@bookedsolid`, and sharing a scope between two distinct projects creates confusion. `@helix` is the correct namespace for HELiX components.

### Why linked versioning?

`@helixui/library` depends on `@helixui/tokens`. A token change (new color, spacing adjustment) must be testable with the exact library version that consumes it. Linked versioning via changesets ensures both packages always publish together at the same version, eliminating "which tokens version works with which library version" confusion.

### Why merge commits only?

Squash commits destroy individual commit history, making `git bisect` unreliable and changelog generation less accurate. Rebase rewrites commit SHAs, breaking cross-references. Merge commits preserve the full feature branch history and produce predictable merge points that changesets and semantic-release can reason about.

### Why human gate on staging-to-main?

Every npm publish has permanent consequences. A broken publish cannot be unpublished after 72 hours (npm policy). For an enterprise healthcare library where consumers build patient-facing applications, the human approval gate on the final promotion to `main` is a deliberate safety mechanism.
